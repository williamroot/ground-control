# #1P — Faturas internas (PDF branded + status) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development`. Steps usam checkbox (`- [ ]`).

**Goal:** fechar o ciclo de billing: a partir de um ciclo de faturamento (#1B), gerar uma **fatura interna** (documento PDF white-label + numeração sequencial por tenant + status `open/paid/overdue/void`), listável/baixável no portal (admin do tenant) e gerenciável no console (agente). Sem emissão fiscal/contábil nesta fase.

**Architecture:** tabelas `invoice` + `invoice_line` (tenant-scoped, RLS). Domain `invoice_service.py` constrói a fatura a partir dos `consumption_event` do período do ciclo (#1B), numera sob lock por tenant, e renderiza PDF com **WeasyPrint** (HTML/CSS com logo+cores do tenant). PDF guardado como `bytea` na própria linha (transacional, sem coordenação de volume). Worker (#1B) marca `open`→`overdue` ao passar de `due_at`.

**Tech Stack:** FastAPI, SQLAlchemy 2 async, Alembic (RLS), **WeasyPrint** (HTML→PDF), Jinja2 (template), Nuxt 3, pytest, vitest.

**Predecessor de migration:** `down_revision = "0016_ai_generation_log"`.

> **Ponto a verificar (roadmap §3):** WeasyPrint precisa de libs nativas (cairo/pango/gdk-pixbuf/libffi) na imagem do sidecar. Validar no Dockerfile; se inviável, cair para **ReportLab** (Task 4 tem o fallback). Decisão por WeasyPrint = template HTML branded mais limpo.
